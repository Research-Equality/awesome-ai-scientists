# Curation Principles

This repository is intentionally opinionated. These principles explain how that opinion is applied.

## 1. Layered, not flat

We distinguish between:

- Systems
- Platforms
- Modules
- Benchmarks
- Surveys
- Datasets
- Meta-resources

Many repositories collapse these together. This one does not.

## 2. Workflow-aware, not repo-centric

The same ecosystem can be viewed through the research workflow:

- Literature
- Ideation
- Planning
- Data / tool use
- Coding
- Experiment
- Analysis
- Writing

This matters because builders often need to know where a capability sits in the loop, not just what its brand name is.

## 3. Canonical over exhaustive

We prefer the most representative or structurally useful references over endless variants. A smaller, sharper map is more maintainable than a maximal directory.

## 4. Description-first curation

Every entry needs a one-sentence description. If we cannot explain a resource clearly, it probably does not belong yet.

## 5. Tags are part of the curation, not decoration

Tags are not metadata bolted on afterward. They are part of the editorial model that makes the map comparable across categories.

## 6. Official-source-first

Whenever possible, we use:

1. Official repositories.
2. Official project pages.
3. Official documentation.
4. Official paper pages.

We avoid secondary writeups unless no canonical source exists and the resource is too important to omit.

## 7. Duplicates require justification

If the same resource appears in more than one section, the duplicate placement must illuminate a different lens of the map, not merely repeat the same link.

## 8. Maintainability beats hype

We prefer:

- Stable links
- Clear function
- Strong explanatory value
- Extendable structure

over trend-chasing or temporary social buzz.

## 9. Honest gaps are acceptable

Some subsections will remain intentionally sparse until the category becomes mature enough to curate well. Placeholder volume is not a quality signal.
